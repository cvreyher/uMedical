# uMedical — MedikamentenProfil.de

A German pharmaceutical data platform that aggregates official regulatory data
about medicines in the EU — and makes it searchable, browsable, and
understandable.

The platform ingests data from the **European Medicines Agency (EMA)** and
pharmacovigilance sources across multiple authorities (**FDA**, **MHRA**,
**Swissmedic**, **BfArM**, **EMA**), links medicines to their active
substances and marketing authorization holders, tracks the full product
lifecycle as a timeline, and monitors safety events and supply shortages —
always with full provenance (every fact knows its source).

## What it does

- 💊 **Medicine profiles** — all EU-authorised medicines with status, ATC code,
  therapeutic indication, designations (orphan, biosimilar, ATMP, …), and
  authorization history
- 🧪 **Substances & companies** — INN-based substance registry and marketing
  authorization holders, linked to their products
- 📅 **Timeline events** — Northdata-style immutable event stream per product:
  authorisations, EPAR updates, withdrawals, safety events
- ⚠️ **Pharmacovigilance** — multi-source safety event aggregation: FDA recalls
  & MedWatch alerts, MHRA safety updates, Rote-Hand-Briefe (BfArM), HPC letters
  (Swissmedic), EMA safety signals
- 📉 **Supply shortages** — shortage tracking across EMA, BfArM, and other
  authorities, linked to products and substances
- 🔍 **Hybrid search (RAG)** — SmPC/EPAR documents are downloaded, chunked
  section-aware, embedded, and searchable via **pgvector (HNSW) + full-text
  (tsvector)** hybrid ranking

## 🧱 Tech Stack

| Layer | Technology |
| --- | --- |
| Monorepo | Turborepo + pnpm workspaces, Node ≥ 22 |
| Backend | NestJS 11 (modular DDD: `ema`, `pvigilance`, `rag`), Pino logging, Swagger/OpenAPI, Zod-validated env |
| Frontend | Next.js 16 (App Router), React 19, TailwindCSS 4, TanStack Query, ECharts/D3 |
| Database | PostgreSQL + **pgvector**, Drizzle ORM (`@workspace/database`) |
| Cache | Redis (keyv) |
| API contract | OpenAPI spec generated by the API → fully typed client via `openapi-fetch` / `openapi-react-query` |
| Storage | Cloudflare R2 (documents/PDFs, optional) |
| Embeddings | OpenAI (`text-embedding-*`, optional) |
| Testing | Vitest (unit), Playwright (e2e), MSW (API mocking) |

## 📦 Project Structure

```bash
├── apps/
│   ├── api/                 # NestJS backend — EMA import, pharmacovigilance feeds, RAG pipeline
│   └── web/                 # Next.js frontend — /medikamente, /wirkstoffe, /unternehmen, /statistiken
├── packages/
│   ├── database/            # Drizzle ORM schemas, migrations & seed scripts
│   ├── eslint-config/       # Shared ESLint configuration
│   ├── icons/               # SVG icon library with React components
│   ├── typescript-config/   # Shared TypeScript configurations
│   └── ui/                  # Shared React component library (shadcn/ui)
├── docker/                  # Local infrastructure (Postgres + pgvector, Redis)
├── pnpm-workspace.yaml
└── turbo.json
```

The API is organized in three business modules, each split into
`application` / `infrastructure` / `presentation`:

- **`ema`** — imports the EMA medicines/companies/substances dataset and serves
  medicines & statistics endpoints
- **`pvigilance`** — polls regulatory feeds (RSS/APIs), normalizes them into
  pharmacovigilance events with product links
- **`rag`** — document pipeline (download → R2 → PDF extraction → chunking →
  embeddings) and hybrid search

## 📖 Getting Started (Local Development)

### 1. Install Dependencies

```bash
pnpm install
```

### 2. Configure Environment Variables

```bash
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env
cp packages/database/.env.example packages/database/.env
```

The defaults work out of the box with the Docker infrastructure below. Note
`LIVE_FETCH_ENABLED=false` in `apps/api/.env` — see [Live data vs. seeded data](#live-data-vs-seeded-data).

### 3. Start Infrastructure (Docker)

All required services run in Docker: Postgres with **pgvector** (host port `5433`)
and Redis (host port `6380`) — non-default ports so they don't clash with other
local projects.

```bash
pnpm infra:up      # start Postgres + Redis (waits until healthy)
pnpm infra:down    # stop services (data is kept)
pnpm infra:reset   # stop, DELETE all data volumes, start fresh
```

### 4. Run Database Migrations

```bash
pnpm db:setup      # builds the database package, then runs migrations
```

> After changing a schema in `packages/database/src/schemas/`, run
> `pnpm --filter @workspace/database build && pnpm --filter @workspace/database db:generate`
> and then `pnpm db:setup` again. (drizzle-kit reads the compiled `dist/` output, so the build step is required.)

### 5. Seed Local Development Data

```bash
pnpm db:seed:dev
```

This fills the database with realistic sample data (medicines, substances,
companies, timeline events, pharmacovigilance events, shortages) so you can
design and develop the website **without calling any live APIs**. The script is
idempotent — run it as often as you like.

### 6. Start the Development Servers

```bash
pnpm dev
```

- API: http://localhost:3000 (Swagger UI at `/api`, OpenAPI spec at `/openapi.yaml`)
- Web: http://localhost:8000

### Live data vs. seeded data

The API has background cron jobs that poll external services (EMA, FDA, MHRA,
BfArM, Swissmedic feeds; EMA document downloads; OpenAI embeddings). Hitting
those constantly during development can get you **rate-limited**, so they are
controlled by a single flag in `apps/api/.env`:

```bash
LIVE_FETCH_ENABLED=false   # default for local dev: no background fetching, use seeded data
LIVE_FETCH_ENABLED=true    # enable background feed polling + job processing (production behaviour)
```

When disabled, the API logs a warning at startup and all background fetch/job
crons are skipped. Manual admin endpoints (e.g. triggering an EMA import via
Swagger) still work, but queued processing jobs only run while the flag is `true`.

### Typed API client

The web app consumes the API through a fully typed client generated from the
API's OpenAPI spec. After changing controllers/DTOs in the API (with the API
running):

```bash
pnpm --filter web generate:api   # regenerates apps/web/src/types/openapi.d.ts
```

### Testing

```bash
pnpm --filter api test           # API unit tests (Vitest)
pnpm --filter api test:e2e       # API e2e tests
pnpm --filter web test:unit      # Web unit tests (Vitest + Testing Library + MSW)
pnpm --filter web test:e2e       # Web e2e tests (Playwright)
```

## 🧭 Design Principles

- **One database for everything** — structure, full-text and vector search live
  in Postgres (pgvector + tsvector); no separate search engine or graph DB
- **Provenance first** — every imported fact and event carries its source URL,
  source type, and confidence; essential for medical data
- **EMA is the leading source** — other authorities contribute events/incidents
  only, linked via INN, without duplicating product data
- **Incremental ingestion** — feeds are polled on individual intervals with
  health tracking and exponential backoff
